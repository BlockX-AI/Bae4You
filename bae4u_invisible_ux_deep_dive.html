What a wallet actually is
A wallet is just a private key + public address
Nothing more. The wallet is a pair of cryptographic keys — a private key (secret, signs transactions) and a public address (visible, receives funds). "Creating a wallet" means generating this key pair. Everything else — the UI, the balance display, the transaction history — is software built on top of these two strings.

Why custodial-only is dangerous for Bae4U
Full custodial = you hold all the keys = you carry all the risk
If Bae4U holds users' private keys in a database and that database is breached, every user's funds are gone. This is exactly what happened to FTX and Celsius. It also triggers Money Services Business (MSB) licensing requirements in the US and similar in the EU. For a dating app, this is not a bet worth taking. The solution is embedded wallets using MPC (Multi-Party Computation) where no single party ever holds the complete key.
What MPC (multi-party computation) actually means
The private key is never assembled in one place
In MPC, the private key is split into "key shares" distributed across multiple parties — typically the user's device, the wallet provider's server, and optionally a third backup party. To sign a transaction, the parties collaborate mathematically to produce a valid signature without any single party ever seeing the complete key. The key is never assembled. It cannot be stolen from any single server breach.

What "smart wallet" adds on top of MPC
Batched transactions
A smart wallet (ERC-4337 account abstraction) can batch multiple blockchain calls into a single transaction. When a user buys a pet, the approve + buy calls happen in one transaction instead of two. Halves the number of user-visible steps from zero to even-more-zero.
Sponsored transactions (paymaster)
Smart wallets can delegate gas payment to a paymaster contract. This is what makes "user never pays gas" possible. The smart wallet calls the paymaster, which pays the miner/validator on the user's behalf. The platform pays the actual ETH cost in bulk, outside the user's view.
What a paymaster is
A smart contract that pays gas on behalf of users
ERC-4337 introduced the paymaster — a contract that intercepts a user's transaction request, verifies it meets the platform's sponsorship rules, and covers the ETH gas cost. From the user's perspective: they tap "Buy Pet", it processes. No ETH balance required. No gas prompt. No "insufficient funds for gas" errors.

Paymaster sponsorship rules (what Bae4U sets)
Whitelist by contract
Only sponsor gas for calls to PetsMarket.sol, PetsCash.sol, and PetsRanking.sol. If a user's wallet tries to call any other contract, the paymaster refuses. This prevents abuse — users cannot use Bae4U's gas budget to do arbitrary blockchain transactions.
Rate limit per user
Maximum 50 sponsored transactions per user per day. Above that, the user must hold a small ETH balance (optional) or the transaction queues. In practice, 99% of Bae4U users will never hit this limit — it protects against bots and gas draining attacks.
Function-level filtering
The paymaster can decode the transaction calldata and only sponsor specific function signatures — e.g. buy(), claimBonus(), lockPet(). More granular than contract-level whitelisting.
Budget caps
Set a daily ETH budget cap in the paymaster dashboard (e.g. $500/day). Once hit, the paymaster stops sponsoring until manual top-up or reset. Essential for early stage when gas costs could spike unexpectedly.

What an on-ramp does
Converts real money into crypto, deposited to the user's wallet
An on-ramp provider handles the regulatory compliance (KYC/AML), the banking relationships, the card processing, and the crypto delivery. Without them, Bae4U would need a money transmitter licence in every country it operates. With them, Bae4U gets a widget SDK and a webhook — the provider handles everything else.

Provider routing logic in Bae4U backend
Auto-routing based on user location and payment method
Backend detects user country from IP + KYC data. India → Transak first (UPI). US/EU with card → MoonPay first (better rates). Other emerging markets → Ramp first. Hard engineering and UX challenges
Key recovery when user loses their device
With MPC embedded wallets, key shard A lives on the device. If the user loses their phone, shard A is gone. Recovery relies on shard B (CDP server) + a backup shard the user set up during onboarding — typically an email-based recovery or a passkey. If the user skipped backup setup (common in low-friction onboarding), recovery becomes very difficult. Solution: make recovery setup a first-session prompt with a simple passkey flow. Never let users get to a "your funds might be stuck" situation.
Network latency makes "invisible" feel slow
On Ethereum mainnet, block times are 12 seconds. On Base L2, it's 2 seconds. But the full confirmation cycle (user action → bundler → paymaster check → block inclusion → event confirmed) can take 5–15 seconds on a slow connection. A dating app user expects tap → immediate response. Solution: optimistic UI updates. Treat the transaction as successful the moment it hits the bundler, update the UI immediately, and silently roll back if it fails (rare). The user never sees "waiting for confirmation."
Paymaster budget exhaustion during viral moments
If Bae4U gets featured on a popular creator's page and 50,000 users sign up in an hour, the paymaster can burn through its daily ETH budget in minutes. New users then get "transaction failed" errors that have nothing to do with them — creating a catastrophic first impression. Solution: dynamic budget scaling tied to user growth rate, with Slack alerts when budget reaches 30% remaining. Pre-fund the paymaster with 2 weeks of buffer at projected growth.
On-ramp KYC friction drops conversion
Transak and MoonPay require ID verification for purchases above certain thresholds (typically $150–200). This verification takes 2–5 minutes and requires a photo ID and selfie. For a dating app user who just wanted to buy a pet, this is jarring. Solution: keep first-purchase amounts below KYC thresholds (offer $5, $10, $20 starter packs). Gate the KYC prompt only when the user explicitly tries to buy more. Most users will never need KYC on day one.
EIP-712 signature verification is subtle to implement correctly
The login bonus uses EIP-712 structured data signing. If the domain separator, type hash, or chainId is misconfigured, users get a "signature verification failed" error on their bonus claim — with no explanation. This is invisible to users (they just see "bonus not available") but silently hemorrhages engagement. Solution: build a separate test harness that runs 1,000 sign-and-verify cycles across all device types before shipping. Make the error logging verbose on the contract level during testnet.
Multi-device sync — two phones, one wallet
A user installs Bae4U on their phone and their tablet. MPC shard A lives on both devices, but they may be different shards generated separately. The wallet address must be the same on both devices. CDP's MPC model handles this via the server shard (shard B) being the consistent party — when the user logs in on a new device, a new shard A is generated and bound to the same shard B. Test this path heavily. Users switching phones is common.
Fiat currency volatility between widget open and transaction confirm
A user opens the "buy $20 of Gold" widget. ETH price moves 3% while they complete KYC. They end up with slightly more or less ETH than displayed. On Transak, the price lock lasts only a few minutes. Solution: show the ETH amount prominently in the widget and refresh the quote every 30 seconds. Add a "price may vary slightly" disclosure. Never show a fixed ETH amount without a refresh mechanism — it creates support tickets.
Complete invisible UX journey — signup to first successful trade
1
User taps "Sign up with Google"
CDP SDK intercepts. OAuth flow completes. Behind the scenes: key shard A generated on device, key shard B created on CDP servers. MPC coordination establishes the wallet address. An ERC-1155 SFT is minted by PetsRegistry.sol. The user sees: a profile setup screen. They have no idea a wallet was created.
2
User fills in profile — app shows "Your profile is live!"
Backend creates the user DB record, indexes the personality vector in Pinecone, and the SFT metadata (avatar IPFS hash, token URI) is pinned. A BullMQ job fires to set the "first-owner" bonus window — the next person to buy this user gets an extra reward. Still no wallet prompt, no blockchain mention.
3
Five minutes later: push notification — "Someone added you to their collection!"
Another user ran buy(tokenId) on PetsMarket.sol. The 10% price invariant fired. Profit split happened automatically. The user's Pets Cash balance increased. The backend indexed the PetPurchased event from The Graph, translated "profit_to_pet_wei" into a displayable Pets Cash amount, and sent the push notification. The user sees a number. Not wei. Not ETH. A clean balance like "you earned 120 Pets Cash."
4
User opens Pets tab, taps on someone's profile, taps "Add to collection"
The UI says "Adding to collection..." with a spinner. Behind the scenes: the CDP SDK constructs a UserOperation (ERC-4337). The bundler picks it up from the alt mempool. The paymaster contract checks it — this is a whitelisted buy() call, budget is available, user is under rate limit — and signs the sponsorship. The bundler submits. 2 seconds later on Base: confirmed. The user's portfolio view updates. The previous owner gets a notification. All without a single confirmation prompt.
5
User logs in 4 hours later — sees "Daily bonus available"
Backend checks last_login_at against the on-chain bonus_claimed_at timestamp. Eligible. Backend's EIP-712 signer service generates a signed bonus claim: a structured message containing the user's address, bonus amount, nonce, and expiry. This is sent to the app. The SDK constructs a UserOperation calling claimBonus() with the signed claim. Paymaster sponsors. PetsCash.sol mints the bonus. Balance updates. The user tapped one button that said "Claim daily bonus" — no signing, no gas, no waiting.
6
User wants to buy a more expensive pet — not enough Pets Cash
UI shows a "Get more Pets Cash" button. Tapping it opens the Gold purchase modal. The backend selects Transak (user is in India, so UPI is available). Transak SDK renders in-app. User enters ₹500, selects UPI, authenticates with UPI PIN. Transak processes the payment, converts to ETH, sends ETH to the user's embedded wallet. Backend receives the Transak webhook confirming the payment, records it in fiat_transactions table. A separate job converts the ETH to Pets Cash via the Gold redemption contract. The user's balance updates. Total time: under 90 seconds. Total friction: entering a UPI PIN, same as any app purchase.
