Quality over quantity. Security by design.
Verified profiles, gender-balanced mechanics, decentralised identity, and an anti-harassment architecture. Built for people who are done with the race to the bottom of legacy dating apps.

08 · Architecture Principle
Economic truth
on-chain. Everything else, off.
The most consequential architectural decision. Get it wrong and you have a $5-per-click product — or just a regular app with a wallet bolted on.
On-Chain — Economic Truth
User Profile SFTs (ERC-1155)
Every user minted as a Semi-Fungible Token on signup. Token ID → user, price, owner, lock status, ghost flag.
Pets Market — buy/sell/10% rule
The 10% invariant lives here. Not in a database. Smart contract enforces it — trustless, unhackable.
Pets Cash Token (ERC-20)
All transfers on-chain. Balances publicly verifiable — required for ranking system to be trustworthy.
Badge SFTs — weekly issuance
On-chain minting = truly owned, verifiable credentials. Not participation trophies in a database.
SFT Creator Passes
Ownership record on-chain. Token-gates off-chain content access. Transferable, resellable by holders.
Lock status + expiry timestamp
Contract-enforced lock. If it lived in your DB, a compromised backend could delete it and steal a pet.
Off-Chain — Everything Else
Wish List
Pure user intent. No economic consequence until a buy. Paying gas to bookmark a profile is absurd.
Ranking Computation (The Graph)
O(n) reads on-chain = prohibitive at scale. Compute off-chain via subgraph, verify result on-chain.
Login Bonus Triggering
EIP-712 signed claims. Backend signs, user submits to contract. Trustless without direct backend authority.
Profile content (photos, bio)
Storage-heavy. On-chain storage at $1+ per "like" kills any social app. Use IPFS or CDN.
Matching algorithm
18-dimension personality scoring. Computational, no trust requirements. Traditional backend.
Chat, notifications, search
Real-time, high-frequency, zero financial consequence. On-chain messaging is catastrophically bad UX.

The hybrid layer: Rankings are computed off-chain (The Graph indexes all PetPurchased events) but badge SFT minting happens on-chain via backend-signed proofs. Computed off-chain. Verifiable by anyone. Anchored on-chain.

09 · Smart Contract Architecture
Four contracts.
One economy.
Contract 01
PetsRegistry.sol
Mints user SFT on account creation. Sets initial price. Maintains user → token ID mapping. The entry point for all new participants in the economy.
Contract 02
PetsMarket.sol
Core trading contract. buy(tokenId) enforces the 10% price increase, splits profit 50/50 automatically, checks lock status, emits PetPurchased events.
Contract 03
PetsCash.sol (ERC-20)
In-game currency token. Minted on login bonuses (via EIP-712 signed claims). Transferred on all purchases, splits, and gifts. Bridgeable to Gold.
Contract 04
PetsRanking.sol
Reads Assets on-chain. Weekly snapshot via Chainlink Automation. Issues badge SFTs on threshold crossing. Executes weekly standing resets.
The Core buy() Function
function buy(uint256 tokenId) external { Pet storage pet = pets[tokenId]; // Guard: lock + balance checks require(!pet.locked, "Pet is locked"); require( petsCash.balanceOf(msg.sender) >= pet.price, "Insufficient balance" ); address prevOwner = pet.owner; address petOwner = pet.userAddress; uint256 salePrice = pet.price; // Calculate profit split uint256 cost = pet.purchasePrice[prevOwner]; uint256 profit = salePrice - cost; uint256 half = profit / 2; // Settle payment petsCash.transferFrom(msg.sender, address(this), salePrice); petsCash.transfer(prevOwner, cost + half); petsCash.transfer(petOwner, half); // Update state pet.owner = msg.sender; pet.purchasePrice[msg.sender] = salePrice; // THE INVARIANT — 10% price increase pet.price = (salePrice * 11000) / 10000; emit PetPurchased(tokenId, prevOwner, msg.sender, salePrice, pet.price); }
10 · Engineering Challenges
The problems
nobody warns you about.
Challenge 01 — Oracle Problem
Login Bonuses
Off-chain event (login) must trigger on-chain state (mint Cash). Solution: EIP-712 signed claims. Backend signs the bonus. User submits signature to contract. Contract verifies — not trusts — the backend. Never have your backend call mintBonus() directly with an owner key — that's a centralised failure surface.
Challenge 02 — Gas UX
Every transaction has friction
Buy, sell, lock, gift — all on-chain. At mainnet gas prices, a single pet purchase could cost more than the pet itself. Solution: Deploy on L2 (Base or Polygon). Use a paymaster contract to sponsor gas fees for users. Abstract wallets via Coinbase embedded wallet SDK. The user should never see the words "gas fee."
Challenge 03 — Rankings at Scale
O(n) reads are lethal
Computing rankings fully on-chain means iterating all users' assets — prohibitively expensive at scale. Solution: The Graph protocol indexes PetPurchased events in real-time. Off-chain computation builds rankings. Backend submits signed proof. Contract mints badge SFT. Rankings are verifiable by anyone — the on-chain data is public.
Challenge 04 — Ghost Pets
Deactivation without deletion
When a user deactivates, their SFT can't disappear — it's still in someone's portfolio affecting their Assets. Implement as a status enum: PetStatus { Active, Ghost, Burned }. Ghost = price frozen, non-transferable, but asset-counted. Deactivation requires a user-signed transaction to prevent backend abuse.
Challenge 05 — Lock Mechanic
Self-expiring, no manual unlock
Use block.timestamp comparison. Auto-expire when block.timestamp >= lockExpiry. Locks self-resolve without a manual unlock call — one less transaction, cleaner UX, no forgotten locks.
Challenge 06 — Rounding
Document before you code
Integer division truncates. On a sale of ₹1,001, profit / 2 rounds down — one satoshi disappears. At millions of transactions this becomes a systemic accounting error. Use basis points throughout: price × 11000 / 10000. Define your rounding policy before writing a single function.
11 · Competitive Intelligence
What came before.
What we learned.
Three precedents that prove the model, warn of failure modes, and define the design space Bae4U occupies.
SocialFi · 2023
Friend.tech
$230M+ Volume · 200K users
People-as-tradeable-assets. Users' social influence represented as Keys (shares). Built on Base. Paradigm-backed at $50M seed. Proved the model works at launch.
Failure: Trading peaked on day of creation, dropping 80% by Day 2. No utility beyond speculation. Top 0.1% captured 47.9% of earnings. Bae4U's fix: dating utility independent of token price, plus tiered rankings so new users have winnable competitions.
Blockchain Gaming · 2021
Axie Infinity
2.7M daily users → 250K after crash
Play-to-earn Pokémon-style game. $3B valuation at peak. Proved blockchain gaming could reach mainstream scale. Inspired entire P2E category.
Failure: SLP token crashed 99% when new player inflow slowed — classic Ponzi mechanics. No token sinks meant hyperinflation. Bae4U's fix: multiple token sinks (premium features, verified badges, SFT passes) absorb Pets Cash to control supply.
Blockchain Dating · 2022
Lonely Ape Dating Club
Cancelled before launch
NFT-gated dating app for BAYC holders. Featured wallet-based matching, crypto net worth filters, match-to-earn. Cancelled within months.
Failure: TAM of 6,400 BAYC holders — overwhelmingly male, many not single. No viral mechanic. No retention layer. Bae4U's fix: open to all users, Pets game drives retention regardless of dating success, identity verification solves the catfish/authenticity problem.

The synthesis: Friend.tech proved people-as-assets works. Axie proved blockchain gaming retains users. Lonely Ape proved dating-only crypto fails. Bae4U is the first platform to combine all three insights — social utility + game retention + open access — into a single architecture.

13 · Competitive Moats
Five things that
make this survive.
90% of Web3 projects fail within 18 months. The survivors share five structural properties. Here's how Bae4U maps against each one.
1
Real utility beyond speculation
The fatal mode: product only works when token price goes up. Bae4U's fix: if Pets Cash crashes, users stay because the dating app is genuinely good. If matches are slow, the Pets game is still addictive. The three layers provide mutual retention insurance — something no pure SocialFi app has ever achieved.
2
Token sinks prevent hyperinflation
Pets Cash has six minting sources (inflation). It needs sinks: premium verification badges, profile boosts in discovery, advanced personality analysis, ranked tournament entry, SFT pass purchases denominated in Pets Cash. Without sinks, the currency inflates → users feel poorer → they leave. Design sinks before mainnet.
3
Invisible UX — blockchain as backend
A dating app user should never see the words "gas fee," "smart contract," or "seed phrase." Auto-create custodial wallets on signup. Sponsor gas via paymaster. Show Pets Cash as an app balance. Make "buy this pet" feel like "add to cart." The blockchain is your infrastructure — as invisible to users as AWS is to Netflix subscribers.
4
Tiered competition prevents whale dominance
Friend.tech's top 0.1% captured 47.9% of all earnings — a wealth concentration that kills social products. Country rankings, weekly resets, and activity-bracket tiers ensure new players always have winnable competitions. Remove global leaderboard dominance as a discouragement vector.
5
Compliance identity as a dating safety moat
Dating apps have an unsolved identity crisis: catfishing, bots, predatory accounts. On-chain verified identity (TokenProof) creates a verified-profile layer no legacy dating app can replicate. A user with verified on-chain identity, 6-month tenure, 40 pets, and rising value is provably a real, engaged human. That's Bae4U's Tinder-killing moat.
14 · Risk Matrix
Know the risks
before you ship.
Technical Risks
Critical
Smart contract exploit in profit split
Integer overflow, rounding errors, or re-entrancy in the 50/50 split logic. Mandatory: two independent audits, formal verification of the buy() function, bug bounty before mainnet.
High
Gas UX friction kills mainstream adoption
Abstract entirely via embedded wallets and paymasters. Never surface gas to the user. Test onboarding with Web2-native users before launch.
Medium
Bot manipulation of Pets market
Bots can front-run new user mints (first-owner bonus). Rate limiting on buys, anti-bot off-chain detection layer, cooldown periods on first-owner claims.
Medium
Regulatory classification of Pets Cash
Does Pets Cash constitute a security in target markets? Get a legal opinion before mainnet launch. Structure it as a utility token with clear in-app purpose.
Product & Market Risks
High
Token price crash kills engagement
Ensure dating utility is fully independent of Pets Cash price. If the token goes to zero, users should still want to use the dating features. Design for this scenario explicitly.
High
Whale concentration dominates rankings
Tiered rankings by country, activity bracket, and tenure. Cap global leaderboard impact. Weekly resets are necessary but not sufficient alone.
High
Dating safety and harassment vectors
The Pets mechanic (buying people) can feel dehumanising without careful framing. Strong opt-out controls, community moderation, and the identity verification layer are the mitigation.
Medium
Value inflation locks out new players
Top pets become exponentially expensive. Add value decay curves or tier-based markets so new users have accessible, affordable entry points even after the platform matures.
